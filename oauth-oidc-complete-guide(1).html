Section 01 · Foundations

History: Why Was OAuth Invented?

To understand OAuth, you must understand the problem it solved — the Password Anti-Pattern: before OAuth, the only way a third-party app could access your data on another service was if you gave it your username and password.

Imagine you use a calendar app that wants to import your Gmail contacts. The app says: "Enter your Gmail password." You type it. Now that app holds your Gmail password forever. If the app is compromised, the attacker has your Gmail credentials. You can't revoke access without changing your password, and the app has full account access — not just contacts.

The Core Problem

Sharing passwords gives all-or-nothing access, no expiration, no revocation, and zero audit trail. A breach of the third party = your password leaked.

Timeline: The Road to Modern Identity

2006–2007
The Problem Becomes Acute
Twitter, Flickr, and early social APIs explode. Developers need to access user data on their behalf. Password sharing is the universal (horrible) solution.
Sep 2007
OAuth 1.0 Discussion Begins
Blaine Cook (Twitter), Chris Messina, and others start an open working group. Core concept: give third parties a token, not a password. Tokens can be scoped and revoked.
Apr 2010
RFC 5849 — OAuth 1.0 Published
OAuth 1.0a becomes an IETF RFC. Used cryptographic HMAC-SHA1 signatures on every request. Functional but complex and painful for developers to implement correctly.
Oct 2012
RFC 6749/6750 — OAuth 2.0
Complete redesign. Dropped signatures in favor of TLS. Introduced grant types to support different architectures. Became the dominant industry standard for API authorization.
Feb 2014
OpenID Connect 1.0
The OpenID Foundation built an identity layer on top of OAuth 2.0, adding the ID Token (JWT) and standardized user info. Authentication + Authorization finally unified.
2017–2019
Security Hardening
PKCE (RFC 7636) published. Implicit Grant officially deprecated. OAuth 2.0 Security BCP drafted. The ecosystem learns from its mistakes.
2022–Present
OAuth 2.1 & Modern Security
DPoP standardized (RFC 9449). mTLS client auth (RFC 8705) widely adopted. OAuth 2.1 consolidates best practices. FAPI 2.0 for financial-grade APIs.

What Did OAuth Replace?

🔑

Password Sharing (Primary Target)

Apps asking users for passwords to third-party services. Zero security properties. This was THE problem OAuth directly solved.

🪟

Screen Scraping

Bots logging in as the user and parsing HTML to extract data. Brittle, often ToS violations, no API contract.

🔒

OpenID 1.0 / 2.0

Earlier federated identity (2005–2007). Handled authentication but not authorization. Used complex URLs as identifiers. OAuth 2.0 + OIDC superseded it entirely.

🎫

SAML (Partially)

XML-based enterprise SSO (2002). Still used in enterprise but OIDC has largely replaced it for modern web and mobile apps.

🍪

Proprietary Token Systems

Twitter, Google, Facebook all had their own custom delegation systems before OAuth standardized everything.

🔐

HTTP Basic Auth

RFC 7617. Still used for simple server-to-server but completely inadequate for delegated third-party access or identity federation.

Section 03 · Core Concepts

OAuth 2.0 — The Complete Picture

OAuth 2.0 is defined in RFC 6749 (the framework) and RFC 6750 (Bearer token usage). It is an authorization framework, not an authentication system — a critical distinction.

Critical Distinction

OAuth 2.0 answers: "Can this application do X on behalf of this user?" — It does NOT answer: "Who is this user?" That's what OpenID Connect adds on top.

The Two Required Endpoints

1. Authorization Endpoint — User Interaction

This is where the user logs in and gives consent. The browser redirects here. The client never sees the user's credentials — this is the security foundation of OAuth.

Authorization Request
GET /authorize?
  response_type=code
  &client_id=my-app-id
  &redirect_uri=https://myapp.com/callback
  &scope=openid profile email read:contacts
  &state=xK9f2mP3qR7t         ← CSRF protection, random
  &code_challenge=E9Melhoa2Ow...  ← PKCE challenge
  &code_challenge_method=S256
Host: accounts.google.com

2. Token Endpoint — Server-to-Server

This happens backend-to-backend, invisible to the browser. The client exchanges the authorization code for actual tokens.

Token Exchange Request
POST /token
Host: accounts.google.com
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code=SplxlOBeZQQYbYS6WxSbIA
&redirect_uri=https://myapp.com/callback
&client_id=my-app-id
&client_secret=super-secret     ← server-side only, never in browser
&code_verifier=dBjftJeZ4CVP-mB9...  ← PKCE verifier
Token Response
{
  "access_token":  "eyJhbGciOiJSUzI1NiJ9.eyJzdWIiOi...",
  "token_type":    "Bearer",
  "expires_in":    3600,          ← seconds until access_token expires
  "refresh_token": "8xLOxBtZp8w4iY...",  ← long-lived, store securely
  "id_token":      "eyJhbGciOiJSUzI...",  ← OIDC only, JWT always
  "scope":         "openid profile email read:contacts"
}

Confidential vs Public Clients

This is fundamental. The entire security model differs based on whether a client can securely store a secret.

Client TypeCan Keep Secrets?ExamplesClient Secret?PKCE Required?
ConfidentialYesServer-side apps, backend servicesYesRecommended
PublicNoSPAs, mobile apps, desktop appsNoRequired
Why This Matters

A mobile app's code can be reverse-engineered. Any "client secret" in a mobile app is extractable. Public clients cannot use client secrets — hence PKCE was invented to compensate.

Section 04 · Roles

The Four OAuth Roles

Every actor in every OAuth flow maps to one of four defined roles.

👤

Resource Owner

The entity that owns the protected resource. Usually a human user. Grants delegated access to the client.

Example: You, the Gmail user

📱

Client

The application requesting access. Acts on behalf of the resource owner. Must be registered with the authorization server.

Example: Yelp wanting your contacts

🏛️

Authorization Server

Issues tokens after authenticating the resource owner and getting consent. The trust anchor of the system. May be separate from the resource server.

Example: accounts.google.com

🗄️

Resource Server

Hosts the protected resources (APIs). Validates the access token on each request. Often the same service as the authorization server.

Example: contacts.google.com API

Client Registration

Before any OAuth flow, the client registers with the authorization server:

ValueDescriptionPublic?
client_idUnique app identifier. Included in all requests.Yes
client_secretShared secret for confidential clients only.Server only
redirect_urisExact URIs for redirect. Pre-registration prevents open redirect attacks.Yes
grant_typesWhich grant types this client is allowed to use.Configured
scopesMaximum scopes this client can ever request.Configured
Section 05 · Grant Types

All Grant Types — What They Are & Why They Exist

A grant type defines the mechanism by which the client obtains an access token. Different application architectures require different flows.

Mental Model

Grant types are different roads to the same destination (an access token). Which road you take depends on your application type, whether a user is present, and your security requirements.

Grant TypeRFCUse CaseUser Present?Status
Authorization CodeRFC 6749Server-side web appsYesCurrent
Auth Code + PKCERFC 7636SPAs, mobile, all public clientsYesRecommended
Client CredentialsRFC 6749Machine-to-machine (no user)NoCurrent
Device CodeRFC 8628TVs, CLIs, browserless devicesIndirectCurrent
Refresh TokenRFC 6749Get new access tokens silentlyNoCurrent
JWT BearerRFC 7523Service-to-service with asymmetric keysNoCurrent
~~Implicit~~RFC 6749Was for SPAs pre-PKCEYesDeprecated
~~ROPC~~RFC 6749Direct username/passwordYesDeprecated

Why Implicit Was Deprecated

Implicit returned the access token directly in the URL fragment (#access_token=...). This meant the token ends up in browser history, in server logs via Referer headers, and visible to injected scripts. No refresh token was issued either. PKCE with Auth Code solves all of these.

Why ROPC Was Deprecated

Resource Owner Password Credentials let the client collect the username and password directly and trade them for a token. This defeats the entire purpose of OAuth — the user gives credentials to the third-party app again. It's the password anti-pattern wearing an OAuth costume.

Section 06 · Flows

Authorization Code Flow — Every Step Explained

This is the most important OAuth flow. Every other flow is a variation or simplification of it.

Authorization Code Flow — Complete 9-Step Sequence
1
User → Client
User clicks "Login with Google"
Client generates random state value (CSRF protection) and code_verifier (PKCE). Stores both in session.
2
Client → Browser
Client redirects browser to Authorization Endpoint
302 redirect to accounts.google.com/authorize?response_type=code&client_id=...&state=...&code_challenge=...
3
Browser → Authorization Server
User authenticates with the Authorization Server
Google shows its login page. The client (Yelp) NEVER sees the user's password. This is the core security benefit.
4
Auth Server → User
Consent screen displayed
"Yelp wants to: Read your contacts, View your profile." User clicks Allow. Auth server records this consent grant.
5
Auth Server → Browser → Client
Auth server redirects back with authorization code
myapp.com/callback?code=SplxlOBeZQ&state=xK9f2mP3qR7t — Code is short-lived (typically 60s) and single-use.
6
Client validates
Client verifies the state parameter
If state doesn't match what was sent in step 1, abort immediately — this is a CSRF attack attempt.
7
Client → Auth Server (server-to-server)
Client exchanges code for tokens
POST to /token with code, client_id, client_secret (or PKCE verifier), redirect_uri. Happens backend-to-backend — browser never sees this.
8
Auth Server → Client
Auth server returns tokens
Returns: access_token, refresh_token, id_token (if OIDC), expires_in, scope.
9
Client → Resource Server
Client calls the API with the access token
Authorization: Bearer eyJhbGciOiJSUzI1Ni... on every API request.

Why the Code? Why Not Return the Token Directly?

The authorization code is an intermediate step for a crucial reason: it separates the browser-facing channel from the token delivery channel.

Without Code (Implicit)With Code (Auth Code)
Token in URL → in browser history → in Referer headersCode in URL — harmless without client secret or PKCE verifier
Token accessible to any JavaScript on the pageToken delivered to backend via secure server-to-server call
No way to bind token to specific clientToken bound to client via client_secret or PKCE
Section 07 · Security

PKCE — Proof Key for Code Exchange

PKCE (pronounced "pixie") is defined in RFC 7636. It solves authorization code interception for public clients and is now recommended for ALL clients.

The Attack PKCE Prevents

Without PKCE, a malicious app on the same mobile device can register the same redirect URI scheme (e.g., myapp://callback) and intercept the authorization code. The attacker can then exchange the code for tokens — even though it was meant for the legitimate app.

PKCE Mechanics
1
Generate code_verifier (client only)
A cryptographically random string: 43–128 URL-safe characters. Generated fresh per auth request. Never sent anywhere except the token endpoint.
2
Derive code_challenge
code_challenge = BASE64URL(SHA256(code_verifier)) — a one-way hash. This IS sent publicly in the auth request.
3
Auth server stores challenge with the issued code
When the auth server issues the authorization code, it stores the code_challenge alongside it.
4
Client sends code_verifier in token exchange
Auth server hashes the verifier and compares to stored challenge. Only the original requester knows the verifier — an interceptor who stole just the code cannot complete the exchange.
JavaScript — PKCE Implementation
// Generate code_verifier — cryptographically random
function generateCodeVerifier() {
  const array = new Uint8Array(32);
  crypto.getRandomValues(array);
  return btoa(String.fromCharCode(...array))
    .replace(/\+/g, '-').replace(/\//g, '_').replace(/=/g, '');
}

// Derive code_challenge = BASE64URL(SHA256(verifier))
async function generateCodeChallenge(verifier) {
  const data = new TextEncoder().encode(verifier);
  const digest = await crypto.subtle.digest('SHA-256', data);
  return btoa(String.fromCharCode(...new Uint8Array(digest)))
    .replace(/\+/g, '-').replace(/\//g, '_').replace(/=/g, '');
}

// Usage:
const verifier = generateCodeVerifier();
const challenge = await generateCodeChallenge(verifier);
sessionStorage.setItem('pkce_verifier', verifier);  // store for later

// In auth request URL:
// &code_challenge={challenge}&code_challenge_method=S256

// In token exchange POST body:
// &code_verifier={sessionStorage.getItem('pkce_verifier')}

Client Credentials Flow — Machine to Machine

The simplest OAuth flow. No user is involved. The client authenticates directly with the authorization server using its own identity.

Client Credentials Flow
1
Service A needs to call Service B's API
No human present. Service A is a backend job, microservice, or automated process.
2
Service A POSTs to /token with its credentials
grant_type=client_credentials&client_id=...&client_secret=...&scope=read:orders
3
Auth Server validates and returns access_token
No refresh token — the client can just get a new access token whenever needed.
4
Service A calls Service B with the token
Authorization: Bearer ... on every API request to Service B.
Client Credentials Request
POST /oauth/token
Authorization: Basic base64(client_id:client_secret)
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
&scope=read:orders write:shipments
Common Use Cases

Microservices calling each other, cron jobs, CI/CD pipelines, data sync daemons — any automated process needing API access without a human in the loop.

Section 09 · Flows

Device Authorization Flow

Defined in RFC 8628. For devices that can't open a browser or accept keyboard input: Smart TVs, gaming consoles, IoT devices, CLI tools.

Device Code Flow — The TV Login Pattern
1
Device requests a device_code and user_code
POST to /device/code with client_id and scope. Response includes device_code, user_code, verification_uri, and polling interval.
2
Device shows the user_code on screen
TV shows: "Go to google.com/device and enter code: BDWR-CQPF"
3
Device polls the token endpoint
Every few seconds: POST /token with grant_type=urn:ietf:params:oauth:grant-type:device_code. Gets authorization_pending until user approves.
4
User goes to URL on phone/laptop and enters code
Logs in on their phone, approves. Authorization server marks the device_code as approved.
5
Device's next poll returns the access token
The TV is now authenticated — no browser was needed on the device itself.

Section 11 · JWT

JWT — JSON Web Tokens, Complete Deep Dive

JWT is defined in RFC 7519. It's a compact, URL-safe way to represent claims as a JSON object that can be cryptographically signed and optionally encrypted.

Key Insight

JWT is NOT an OAuth concept. JWT is a general-purpose token format. OAuth can use JWTs, but JWTs exist completely outside OAuth. OIDC ID Tokens are always JWTs. OAuth Access Tokens may or may not be JWTs.

JWT Structure — Three Parts

A JWT has exactly 3 parts separated by periods. Each part is Base64URL-encoded (not Base64 — URL-safe variant):

eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCIsImtpZCI6Imt5LTIwMjQtMDEifQ.eyJpc3MiOiJodHRwczovL2FjY291bnRzLmdvb2dsZS5jb20iLCJzdWIiOiIxMTg0MDk5OTMxNTg5NjA4OTEwNzciLCJhdWQiOiI0MDc0MDg3MTgxOTIiLCJleHAiOjE3MTYyMzkwMjIsImlhdCI6MTcxNjIzNTQyMiwiZW1haWwiOiJ1c2VyQGV4YW1wbGUuY29tIiwibmFtZSI6IkphbmUgRG9lIiwibm9uY2UiOiJhYmMxMjMifQ.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c_anythingHereIsSignature
Header: algorithm & key ID
Payload: claims (your data)
Signature: cryptographic proof

Header

JWT Header (Decoded)
{
  "alg": "RS256",     ← Algorithm used to sign (RS256, ES256, HS256...)
  "typ": "JWT",      ← Token type
  "kid": "ky-2024-01" ← Key ID — tells verifier which public key to use from JWKS
}

Payload — Three Types of Claims

JWT Payload — All Claim Types Annotated
{
  // REGISTERED CLAIMS (RFC 7519 standard — all optional but highly recommended)
  "iss": "https://accounts.google.com",
         ↑ Issuer — who created this token. Must be HTTPS URL in OIDC.

  "sub": "1184099931589608910",
         ↑ Subject — unique, stable user ID. Not an email. Never reassigned.

  "aud": "407408718192.apps.googleusercontent.com",
         ↑ Audience — who this token is FOR. Recipient MUST verify they're in aud.

  "exp": 1716239022,   ← Expiration time (Unix epoch seconds). MUST verify.
  "nbf": 1716235422,   ← Not Before — token invalid before this time.
  "iat": 1716235422,   ← Issued At — when was the token created.
  "jti": "unique-id-xyz", ← JWT ID — unique per token. Use for replay detection.

  // PUBLIC CLAIMS (OIDC standard scope-based)
  "name":           "Jane Doe",
  "email":          "jane@example.com",
  "email_verified": true,
  "picture":        "https://...",
  "nonce":          "n-0S6_WzA2Mj",  ← OIDC: replay prevention

Why Asymmetric Signing Matters

With RS256/ES256: the auth server signs with its private key. Anyone can verify using the published public key (from the JWKS endpoint). The resource server can verify tokens locally without any network call to the auth server — critical for performance at scale.

How Signature Verification Works
// Auth server creates signature:
signature = RSA_SHA256_Sign(
  base64url(header) + "." + base64url(payload),
  privateKey   ← Only the auth server has this
)

// Resource server verifies (no private key needed!):
isValid = RSA_SHA256_Verify(
  base64url(header) + "." + base64url(payload),
  signature,
  publicKey    ← Fetched from JWKS endpoint and cached
)
Section 12 · Security

JWT Security — Validation, Pitfalls & Attacks

Complete JWT Validation Checklist

Every application receiving a JWT MUST validate ALL of these:

JWT Validation — All Steps Required
1
Parse structure
Exactly 3 parts separated by dots. Each part must be valid Base64URL. Header must be valid JSON.
2
Verify algorithm is allowed
Check alg in header matches your allowlist. NEVER accept none. NEVER dynamically accept any algorithm from the header.
3
Fetch correct public key
Use kid from header to look up the right key from JWKS. Cache JWKS; re-fetch only on unknown kid.
4
Verify the signature
Cryptographic verification over header+payload using the public key. If this fails, reject immediately. Do not inspect claims.
5
Check exp (expiration)
now < exp. Allow small clock skew (30–60s). Reject expired tokens.
6
Check iss (issuer)
Must exactly match expected issuer URL. Prevents tokens from one service being used at another.
7
Check aud (audience)
Must contain your client_id or API identifier. Prevents token substitution attacks across services.
8
Check nbf (not before) if present
now >= nbf. Reject if current time is before the token's valid-from time.
9
Check nonce (OIDC ID Tokens only)
Must match the nonce from the original authorization request. Prevents replay attacks on ID tokens.

The alg:none Attack

Critical Vulnerability — alg:none

Some early JWT libraries accepted "alg":"none" with no signature. An attacker could forge any claims. The spec technically allows unsigned tokens for trusted channels — but this was widely misimplemented. Fix: always specify your allowed algorithms explicitly, never accept none in production.

The Fix — Always Specify Allowed Algorithms
// WRONG — library may accept "none" dynamically
jwt.verify(token, publicKey)

// CORRECT — allowlist only specific algorithms
jwt.verify(token, publicKey, { algorithms: ['RS256'] })

// Also correct for multiple valid algorithms
jwt.verify(token, publicKey, { algorithms: ['RS256', 'ES256'] })

RS256 → HS256 Confusion Attack

If a server normally uses RS256 (asymmetric), an attacker might send a token signed with HS256 using the server's own public key as the HMAC secret. A vulnerable library told to accept "any" algorithm would use the public key as the HMAC verification key — and the verification would succeed because the attacker signed with it. Again: always specify the exact expected algorithm.

Section 13 · Tokens

Refresh Tokens — Lifecycle & Security

Refresh tokens allow clients to obtain new access tokens without requiring the user to re-authenticate. They are long-lived credentials and must be protected as carefully as passwords.

Why Refresh Tokens Exist

Access tokens are short-lived (5–60 min) to limit misuse if stolen. But making users log in every hour is bad UX. Refresh tokens solve this: they're long-lived (days or months), only used with the authorization server (not APIs directly), and can be rotated to detect theft.

Refresh Token Rotation

Best practice: every time a refresh token is used, it is replaced with a new one. The old one is immediately invalidated.

Why Rotation Detects Theft

If an attacker steals a refresh token, the moment either the attacker or the legitimate user uses it, the authorization server can detect a "double use" and revoke the entire token family. This dramatically limits the impact of a stolen refresh token.

Refresh Token Request
POST /token
Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token
&refresh_token=tGzv3JOkF0XG5Qx2TlKWIA
&client_id=my-app
&client_secret=secret   ← or PKCE verifier for public clients

Section 14 · OIDC

OpenID Connect — Authentication on Top of OAuth

OIDC is a simple identity layer on top of OAuth 2.0. It allows clients to verify who the user is (authentication) in addition to what the app can do (authorization).

The OAuth 2.0 Authentication Problem

OAuth 2.0 alone cannot safely authenticate users. If you receive an access token, all you know is that someone authorized its issuance. You don't know who that someone is, the token format is unspecified, and there's no standard way to verify claims.

Why OAuth Alone ≠ Authentication

Service A gets an access token for Alice from Google. Service A can act as Alice. But if Service A forwards that token to Service B claiming "I'm Alice," Service B can't verify this safely — OAuth access tokens have no standardized audience, issuer, or format. OIDC's ID Token fixes all of this.

What OIDC Adds

🆔

ID Token

A JWT specifically for authenticating the user. Has standardized claims (sub, iss, aud, nonce). Client validates this to prove who the user is.

👤

UserInfo Endpoint

Standard endpoint to get additional profile info beyond the ID token, using the access token.

🔍

Discovery Document

Standardized JSON at /.well-known/openid-configuration describing all endpoints, scopes, and algorithms.

🗝️

JWKS Endpoint

Published public keys in JWK Set format. Clients verify ID token signatures using these without any secret sharing.

📦

Standard Scopes

openid (required), profile, email, address, phone, offline_access.

🔚

End Session Endpoint

Standardized SSO logout — triggers logout across all apps sharing the same identity provider session.

The openid Scope — The Trigger

OAuth vs OIDC
// Pure OAuth (authorization only)
scope=read:contacts write:calendar
→ Returns: access_token, refresh_token
→ You know: "someone granted this" — but NOT who

// OIDC (authentication + authorization)
scope=openid profile email read:contacts
→ Returns: access_token, refresh_token, id_token
→ You know: "Jane Doe (jane@example.com, verified) authorized this"
Section 15 · OIDC

ID Token — Complete Anatomy

The ID Token is the cornerstone of OIDC. It is always a JWT, always signed, and contains verified claims about the authenticated user.

ID Token vs Access Token — The Critical Distinction

PropertyID TokenAccess Token

What does it prove?Who the user is (identity)What the app can do (authorization)

Never Send ID Tokens to APIs

The ID Token's aud is your client_id. If you send it to an API, the API should reject it — it is not in the audience. Use access tokens for API calls. ID tokens are only for your application to establish the user's identity.

All ID Token Claims

ID Token Payload — Fully Annotated
{
  // REQUIRED
  "iss":  "https://accounts.google.com",
         ↑ Must be HTTPS. Client must verify this exactly.

  "sub":  "1184099931589608910",
         ↑ STABLE user ID. Never changes. Never reassigned. Case-sensitive.
           THIS is the true user identity. NOT their email address.

  "aud":  "407408718192.apps.googleusercontent.com",
         ↑ Your client_id. If array, your client_id MUST be in it.

  "exp":  1716239022,    ← Unix epoch. Client must verify not expired.
  "iat":  1716235422,    ← Issued at time.

  // REQUIRED when nonce was sent in auth request
  "nonce": "n-0S6_WzA2Mj",
           ↑ MUST match the nonce from the authorization request. Prevents replay.

  // FROM scope: profile
  "name":       "Jane Doe",
  "given_name": "Jane",
  "picture":    "https://...",
  "locale":     "en-US",
  "updated_at": 1716235000,

  // FROM scope: email
  "email":          "jane@gmail.com",
  "email_verified": true   ← Did the provider verify this email?
}

The sub Claim — The True Identity

The sub claim is the user's permanent, unique identifier within the issuer. Never use email as a primary key — emails change, get transferred, and aren't unique across providers. The combination of iss + sub is globally unique and stable.

Correct User Identification Pattern
// WRONG — email can change or be reassigned
user_primary_key = id_token.email

// CORRECT — stable per issuer
user_primary_key = id_token.iss + "#" + id_token.sub
// → "https://accounts.google.com#118409993158960891077"

ID Token vs UserInfo — When to Use Which

Use the ID Token for core identity (sub, email, name) at login time. Use UserInfo when you need additional claims that would bloat the ID token, or when you need fresh profile data later in the session.

Every OIDC-compliant server must publish at: {issuer}/.well-known/openid-configuration

{
  "issuer":                        "https://accounts.google.com",
  "authorization_endpoint":        "https://accounts.google.com/o/oauth2/v2/auth",
  "token_endpoint":                "https://oauth2.googleapis.com/token",
  "userinfo_endpoint":             "https://openidconnect.googleapis.com/v1/userinfo",
  "jwks_uri":                      "https://www.googleapis.com/oauth2/v3/certs",
  "revocation_endpoint":           "https://oauth2.googleapis.com/revoke",
  "scopes_supported":              ["openid", "email", "profile", "..."],
  "response_types_supported":      ["code", "token", "id_token"],
  "grant_types_supported":         ["authorization_code", "refresh_token"],
  "id_token_signing_alg_values_supported": ["RS256"],
  "claims_supported":              ["sub", "iss", "email", "name", "..."],
  "code_challenge_methods_supported": ["S256"]
}

JWKS — JSON Web Key Set

The JWKS endpoint publishes the authorization server's public keys. Clients use these to verify JWT signatures — no out-of-band key exchange needed.

JWKS Response
{
  "keys": [
    {
      "kty": "RSA",           ← Key type
      "use": "sig",           ← Usage: signature (vs encryption)
      "alg": "RS256",         ← Algorithm this key is used with
      "kid": "ky-2024-01",    ← Key ID — must match JWT header kid
      "n":   "pjdss8ZaDfEe...", ← RSA modulus (Base64URL)
      "e":   "AQAB"             ← RSA exponent (65537 in Base64URL)
    },
    {
      // Second key — auth servers always publish multiple keys for rotation
      "kid": "ky-2024-02", ...
    }
  ]
}
JWKS Caching Strategy

Cache the JWKS aggressively (1–24 hours). Only re-fetch when you encounter a kid you don't recognize. This handles key rotation gracefully while avoiding excessive network calls.

Section 20 · Security

Security Best Practices (RFC 9700)

RFC 9700 (OAuth 2.0 Security Best Current Practice) consolidates years of hard-learned lessons.

✅

Always Use PKCE

For all clients — both confidential and public. Even with a client secret, PKCE provides defense in depth against code interception.

✅

Validate state Parameter

Always generate a cryptographically random state, store in session, and verify it's returned unchanged.

✅

Exact Redirect URI Matching

Register exact URIs. Authorization servers must perform exact string matching — never prefix or wildcard.

✅

Short-Lived Access Tokens

15–60 minutes maximum. Shorter is better. Use refresh tokens for long sessions.

✅

Rotate Refresh Tokens

Issue a new one on each use, invalidate the old. Detect double-use (theft indicator). Invalidate the entire token family on detection.

✅

TLS Everywhere

All OAuth endpoints must use HTTPS/TLS. No HTTP for any OAuth endpoint under any circumstances.

🚫

Never Use Implicit Grant

Completely deprecated. Token in URL = token in browser history = token in logs. Use Auth Code + PKCE.

🚫

Never Use ROPC Grant

The client handles the user's password directly. This is the password anti-pattern with extra steps.

The BFF Pattern — Recommended for SPAs

Backend-for-Frontend (BFF): the SPA's backend handles OAuth, stores tokens server-side, and gives the browser only a session cookie (HttpOnly, Secure, SameSite=Strict).

BFF Pattern — Tokens Never Touch Browser JavaScript
1
SPA redirects to BFF backend for login
The BFF server (not browser JS) initiates the OAuth flow.
2
BFF handles the complete OAuth flow server-side
Tokens are stored in the BFF's server-side session. They never reach the browser.
3
BFF sets a secure HttpOnly session cookie
HttpOnly (JS cannot read), Secure (HTTPS only), SameSite=Strict (no CSRF).
4
SPA calls BFF APIs; BFF forwards with token
XSS in the SPA cannot steal tokens — there are none in the browser.

Section 23 · Advanced

Private Key JWT Client Authentication

Instead of a shared client secret, clients authenticate to the token endpoint by signing a JWT assertion with their own private key. No shared secret ever gets transmitted.

Private Key JWT — How It Works
// Client signs this JWT with ITS OWN private key:
{
  "iss": "my-client-id",
  "sub": "my-client-id",
  "aud": "https://as.example.com/token",  ← token endpoint as audience
  "jti": "unique-id",                    ← prevents replay
  "exp": 1716239022                      ← max 5 minutes
}

// Token endpoint request:
POST /token
client_assertion_type=urn:ietf:params:oauth:client-assertion-type:jwt-bearer
&client_assertion=eyJhbGciOiJSUzI1NiIsImtpZCI6Ii4uLiJ9...
&grant_type=client_credentials
&scope=read:data
Why This Is Stronger Than Client Secrets

The private key never leaves the client. The auth server stores only the public key. Compromise of the auth server doesn't expose the client's credentials. Widely used in Google APIs, financial-grade FAPI, and high-security enterprise deployments.

Section 24 · Advanced

Sender-Constrained Tokens: DPoP & mTLS

Standard Bearer tokens are like physical keys — whoever finds them can use them. Sender-constrained tokens cryptographically bind the token to the specific client, making a stolen token useless.

The Bearer Token Problem

Bearer Token Risk

Any entity possessing a bearer token can use it. If stolen via XSS, network interception, log exposure, or storage breach, the attacker can use it freely until expiry.

DPoP — Demonstration of Proof-of-Possession (RFC 9449)

The client generates a key pair and creates a signed "DPoP proof" JWT on each request. The token is bound to the corresponding public key.

DPoP Request
// DPoP Proof JWT header embeds the client's public key:
{ "typ": "dpop+jwt", "alg": "ES256", "jwk": { /* client public key inline */ } }

// DPoP Proof JWT payload is bound to this exact request:
{
  "jti": "unique-proof-id",
  "htm": "GET",                                    ← HTTP method
  "htu": "https://api.example.com/resource",       ← Request URL
  "iat": 1716235422,                             ← Must be very recent
  "ath": "BASE64URL(SHA256(access_token))"        ← Binds proof to token
}

// HTTP request uses both headers:
Authorization: DPoP eyJhbGciOiJFUzI1NiJ9...    ← Token type is DPoP, not Bearer
DPoP:          eyJhbGciOiJFUzI1NiIsImp3a...    ← The proof JWT

mTLS Client Certificate Binding (RFC 8705)

The client presents a TLS client certificate at the token endpoint. The auth server embeds the certificate thumbprint in the token. The resource server verifies the TLS certificate matches the embedded thumbprint on every request.
